Auto Research Final Plan

fleet-dev-first · Endeavour Loop Owner · Thin Experiment Ledger · Post-MVP durable memory adapter split. 실무 검토용 계획 문서 — 섹션 탐색, 방향 비교, 체크 가능한 QA gate를 포함한다.

fleet-dev-first Endeavour = Loop Owner Post-MVP LLM Wiki Adapter Thin Experiment Layer 6 Waves
최종 아키텍처 요약 Architecture
핵심 명제 — 세 가지
  1. fleet-dev는 현재 체크아웃 extension을 직접 로드한다. self-improve 루프의 "수정 → 즉시 검증" 사이클을 무료로 해결한다.
  2. Endeavour carrier는 루프 오너(Loop Owner)다. 전략·가설·keep/discard·pivot 판단을 담당한다. thin layer tool을 통해서만 기록 계층에 접근한다.
  3. Storage는 두 계층으로 분리된다: MVP SSOT인 trial scratch ledger(.fleet/research/)와 future optional durable memory/AAR(.fleet/knowledge/). JobStreamArchive는 연구 SSOT가 아니다.
fleet-dev가 해결하는 것
  • 빌드/배포 없이 extension 변경 즉시 반영
  • PI_FLEET_DEV=1 환경 자동 활성화
  • dev boot / RISEN 컨텍스트 자동 제공
  • self-improve 루프 런타임 완결 가능
fleet-dev가 해결하지 않는 것
  • 재시작 시 JobStreamArchive 소멸 → 디스크 SSOT 필수
  • trial 기록 구조화 (keep/discard/crash)
  • stdout 누적 → context 잠식 방지
  • durable knowledge → Post-MVP LLM Wiki adapter 계층

JobStreamArchive 제약 — fleet-dev 재시작과 결합 시 주의
• 저장소: 프로세스 메모리 → 재시작마다 소멸
• TTL: 3시간 → 장시간 실험에 부적합
• 용량: 8 MB / 2,000-block → truncation 필수
• read-once → 재조회 불가 → 디스크 기록 필수
