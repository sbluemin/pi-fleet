{"id":"fleet-memory-product-doctrine","title":"Fleet Memory product doctrine","body":"# Fleet Memory product doctrine\n\nFleet Memory is a first-class pi-fleet product feature for accumulating verified operational knowledge that helps the fleet perform future missions better.\n\n## Definition\n\nFleet Memory preserves raw source as immutable intelligence and turns it into reviewable markdown/wiki knowledge through patch, audit, approval, log, and rollback flows.\n\n## Non-goals\n\n- It is not hidden automatic AI memory.\n- It is not vector-DB-first RAG.\n- It is not a plain document search feature.\n- It does not make unreviewed knowledge authoritative.\n\n## Core doctrine\n\n- Raw source is the source of truth and should not be edited after capture.\n- Wiki/chart/brief pages are operational knowledge derived from raw source.\n- Schema/doctrine defines how memory may be written, cited, reviewed, and merged.\n- Logs record ingest, approval, rejection, rollback, dry-dock findings, and AARs.\n- Important wiki knowledge must enter through patch queue and require user-visible approval.\n- Context injection must be mission-specific and carrier-specific, not a full memory dump.\n- Fleet Memory must remain file-first, observable, reviewable, and rollbackable.\n\n## pi-fleet fit\n\nFleet Memory matches pi-fleet's Agent Harness model because PI can route memory scouting, drafting, auditing, and review across carrier roles while preserving the Admiral of the Navy's final authority over important knowledge changes.\n\n## Role alignment\n\n- Vanguard maps naturally to source scouting and related-memory search.\n- Chronicle maps naturally to wiki, chart, brief, and AAR drafting.\n- Sentinel maps naturally to security, source-fidelity, and unsafe-memory review when available.\n- Nimitz maps naturally to architectural or doctrinal judgment.\n- Genesis and Ohio consume approved context packs during implementation missions.\n\n## Approval rule\n\nA direct ingest should create a patch.
